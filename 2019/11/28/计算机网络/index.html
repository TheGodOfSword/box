计算机网络泛谈

发表于 2019-11-28 分类于 计算机网络

应用层

  • HTTP 超文本传输协议
  • FTP 文本传输协议
  • SMTP 简单邮件传输协议
  • DNS 域名系统
  • SSH 安全外壳协议
  • DHCP 动态主机配置协议
  • TELNET 远程登录协议

通过应用进程间的交互来完成特定的网络应用
运输层
–

  • TCP 传输控制协议
  • UDP 用户数据报文协议

提供通用的数据传输服务给应用层传送报文。由于一台主机可以同时运行多个线程，因此 多个应用层可以同时使用下面的运输层的服务。

简单的说：
给应用层传输报文用的。并且是可以给多个应用层复用的。

网络层 （ip协议）

  • IP 网络协议
  • ARP 地址转换协议
  • RARP 反向地址转换协议
  • ICMP Internet组管理协议

网络层的任务就是选择合适的网间路由和交换结点， 确保数据及时传送。

网络层把运输层产生的报文段或者用户数据报封装成分组和包进行传送。
在TCP/IP体系结构中因为网络层使用IP协议，因此分组也叫IP数据报，简称IP数据报。

数据链路层

  • ARQ 自动重传协议
  • CSMA/CD 停止等待协议
  • PPP 点对点协议

主机之间的数据传输，是依靠一段段链路完成的。它把ip数据报封装成帧，帧里面包含了必要的控制信息（同步信息，地址信息，差错控制等）。这些控制信息能够使接收端知道一个帧是从哪个比特开始哪个比特结束。 （检错、纠错）。纠错通过可靠性传输协议来进行

物理层

  • 中继器
  • 集线器
  • 网线
  • HUB

在物理层上所传送的数据单位是比特。物理层（physical layer）的作用是实现相邻计算机节点之间比特流的透明传送，尽可能屏蔽掉具体传输介质和物理设备的差异。

ARP协议的工作原理 ？

网络层的协议。
它的作用：完成IP地址与物理地址的映射

原理：

每台主机都在自己的ARP缓冲区会建立一个ARP列表，里面存IP地址和MAC地址的对应关系。

1.源主机发送数据包
2.检查自己ARP列表是否存在该IP地址对应的MAC地址，如果有，直接将数据包发送到这个MAC地址，如果没有，则在就想本地网段发起一个ARP请求的广播包，查询此主机的MAC地址
3.网络中的所有主机收到这个ARP广播后，检查自己IP和发送的IP是否一致，如果不一致就忽略。如果一致就将这个请求的 IP-MAC映射存到ARP列表。并且发给源请求。
4.源请求得到响应也存下来，如果一直没有得到响应就认为没有找到。

TCP的特点

  • 面向连接（像打电话一样，需要接通、挂断）
  • 每个TCP连接只能有两个端点 （一对一）
  • 可靠交付，无差错、不丢失、不重复、按序到达
  • 提供全双工通信。允许通信双方的应用进程再任何时候都能够发送数据。双方都有发送缓存和接收缓存，用来临时存放双方通信的数据
  • 面向字节流传输。

总结一下主要的点：传输数据之前必须建立连接，传输结束后断开连接。使用三次握手来建立连接。在传递数据时，有确认、窗口】、重传、拥塞控制机制。

UDP的特点

  • 无连接
  • 不保证可靠交付
  • 面向报文
  • 无拥塞控制，因此网络出现拥塞不会使源主机的发送速率降低（对实时应用很有用，如 直播，实时视频会议等）
  • 支持一对一、一对多、多对一、多对多
  • 首部开销小，只有8个字节，比TCP的20个字节的首部要短

总结一下主要的点：就是不需要发送数据不需要建立连接，对方不需要给出任何确认。只管发送就完事了。

三次握手

TCP建立连接的过程就做握手，需要三次，所以叫三次握手。

1.双方都处于关闭状态
2.客户端主动打开连接，服务端被动打开连接。并且双方的TCP服务器进程创建传输控制块TCB（可以理解为连接状态控制块）。然后服务端进入监听状态
3.第一次握手：客户端传送 同步位 SYN =1，和一个序号seq = x。(同步已发送)
4.第二次握手：服务端接口到之后，进行第二次握手，SYN = 1,ACK=1 seq=y ,ack = x+1 发送给客户端 状态到达同步收到
5.第三次握手：客户端 ACK=1，seq = x+1,ack=y+1
6.三次握手完成，TCP连接建立完成

总结：双方seq 序列号都是各自定，客户端需要递增一次seq（第三次握手的时候是第一次的递增），第一第二次握手的SYN报文段不能携带数据，应该也算是保证了可靠性传输吧。

为什么两次握手不可以？

假设客户端发送了一个连接请求，服务端收到之后就建立连接（两次握手），可能客户端的这个连接因为网路节点长时间滞留导致延误，已经失效，而此时如果B收到就直接建立了连接了的话，服务端会等待客户端发来数据，导致服务端会浪费大量资源。

为什么不需要四次握手？

完全可靠的通信协议是不存在的。即便第四次、第五次、第n次…… 握手也不能避免调最后一次握手丢包的情况。

Server 端收到 Client 端的 SYN 后，为什么还要传回 SYN？

为了告诉Client端接收到的信息确实就是它发送的信号

传了SYN,为什么还要传ACK

还是围绕可靠性的中心思想，服务端传回SYN表示发送方到接收方的通道没有问题，但是接收方到发送方的通道还是需要ACK信号来验证。

TCP四次挥手的过程

客户端：A
服务端：B
以下用A、B来表示。

第一次挥手：
A的应用进程向其TCP发出连接释放报文段，并停止发送数据。（应用层-> 网络层），A把连接释放报文段首部的终止控制位FIN 置1，序号seq=u，此时A进入FIN-WAIT-1（终止等待1）状态。

第二次挥手：
B收到连接释放的报文段后，立即发出确认，确认的报文包含：ACK=1,seq=v,ack=u+1 （ack是上一步的seq+1）。发出之后，B就进入了CLOSE-WAIT（关闭等待）状态。并且通知应用层进程要关闭连接了，有要发的赶紧发，没要发的就告诉我，我赶紧跟人家确认关闭了。
此时 处于半关闭状态，A已经没有数据发送了，但是B发送的，A仍然要接受。

第三次挥手：
如果B已经没有数据发送给A。那应用层就通知TCP（网络层）释放连接。这时B发送 报文段 FIN=1 ACK=1,seq=w,ack=u+1,确认号跟第二次挥手的要保持一致。这时B进入了最后确认状态

第四次挥手：

A收到B的释放连接报文后，必须对此发出确认。报文段ACK=1,ack=w+1,seq=u+1。然后B收到就进入CLOSED状态。进入时间等待状态，时间等待计时器设置2MSL后，A才进入CLOSED状态。B结束TCP连接的时间早于A。 在进入CLOSED状态后，各方将撤销传输控制块（TCB）

简单理解：
第一次：A要关闭，告诉B。
第二次：B收到了，回复A说，可能还有数据，你可以先等待等待，可以单方面释放连接了。我还要告诉应用层先。
第三次：应用层告诉B 释放TCP连接，B发送给A。
第四次：A收到后告诉B，B立马断开。A还需要等待2MSL时间后关闭

以上 无论是三次握手还是四次挥手，每一次发送报文段，都会消耗序号seq

为什么第二次和第三次挥手不能合并？

因为第二第三次之间，B到A还可能有数据传输。A需要等待B把数据传输完。

TCP是如何保证可靠传输的？

1.应答机制
2.超时重传。TCP发送报文段后，会有启动定时器，等待目的发来确认，如果没收到就会在定时器的时间内重发。
3.数据包校验
4.失序数据包重排序。TCP报文段作为IP数据包来传输，IP数据报可能失序，因此TCP报文段的到达也可能失序，它会把失序数据进行重新排序，再交给应用层
5.丢弃重复数据
6.流量控制。TCP连接每一方都有固定大小的缓冲空间。控制对方发送的流量大小。使用的是滑动窗口协议。
