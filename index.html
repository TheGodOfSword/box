缓存雪崩、穿透、击穿出现的原因及应对办法

发表于 2019-12-01 分类于 redis

缓存雪崩

引起原因：

大面积key失效，请求直接打入DB。导致DB瘫痪。

解决办法：

  1. 设置key随机失效时间，保证失效时间不一样。
  2. 热点Key永不过期，有更新操作就更新缓存
  3. 集群部署，将热点Key均匀分布在不同的节点

缓存穿透

引起原因
大量请求 缓存中没有、DB中也没有。导致DB瘫痪。

解决办法：

  1. 请求参数校验
  2. 网关层Nginx设置ip黑名单
  3. 布隆过滤器，通过较快的数据结构与算法判断缓存中以及DB中是否有数据，如果没有数据就返回，有数据就刷到缓存里面

HTTP状态码

  • 1XX 信息
    100 Continue ：表明到目前为止都很正常，客户端可以继续发送请求或者忽略这个响应。

  • 2XX 成功

  1. 200 OK
  2. 204 No Content ：请求已经成功处理，但是返回的响应报文不包含实体的主体部分。一般在只需要从客户端往服务器发送信息，而不需要返回数据时使用。
  3. 206 Partial Content ：表示客户端进行了范围请求，响应报文包含由 Content-Range 指定范围的实体内容

concurrentHashMap

发表于 2019-11-11 分类于 JAVA并发编程

JDK 1.7 实现方式、组成结构

Segment类型的数组 + ReetrantLock

Segment是一个数组，里面包含有volatile 修饰的HashEntry（真正存放数据的桶），HashEntry的成员变量又有链表，也是volatile修饰的。

综上： 分段数组（分段锁）+链表

Put操作：

1.通过Key的HashCode定位到相应的Segment
2.虽然HashEntry的Value是volatile修饰的，但是并不能保证Put的原子性，所以要加锁
3.尝试获得锁
4.如果获取失败 ，自旋获取，如果失败达到MAX_SCAN_RETRIES次数后阻塞。保证获得锁
5.通过Key的HashCode定位到Segment中的HashEntry
6.遍历该 HashEntry，如果不为空则判断传入的 key 和当前遍历的 key 是否相等，相等则覆盖旧的 value
7.不相等则创建一个新的HashEntry加入Segment中
8.释放步骤1中的锁

类加载器与类加载机制

发表于 2019-10-05 更新于 2019-11-05 分类于 JVM

类加载器

  • JAVA虚拟机自带的加载器
     

    • Bootstrap 根类加载器（启动类加载器），没有父加载器，它负责加载虚拟机的核心类库，如java.lang.等。它的实现依赖于底层操作系统，属于虚拟机实现的一部分，它没有继承java.lang.ClassLoader,*因为它就是最顶层的类加载器**。 C++实现。 （jre/lib/rt.jar）
       

      • Extension 它的父加载器是根类加载器， 如果用户把创建的Jar包放在/jre/lib/ext下，就会自动由该类加载器加载。扩展类加载器是纯java类，是java.lang.ClassLoader的子类 (/jre/lib/ext)，扩展类加载器只加载jar包，需要把class文件打成jar
         

      • System类加载器，应用类加载器。他的父加载器是扩展类加载器，他从环境变量classPath或者系统属性java.class.path所指定的目录中加载类。是用户自定的类加载器的默认父加载器 (java.class.path)

         

  • 类加载器本身也是类加载器，类加载器又是谁加载的呢？？（先有鸡还是现有蛋）

  • 类加载器是由启动类加载器去加载的，启动类加载器是C++写的，内嵌在JVM中。

  • 内嵌于JVM中的启动类加载器会加载java.lang.ClassLoader以及其他的Java平台类。当JVM启动时，一块特殊的机器码会运行，它会加载扩展类加载器以及系统类加载器，（Launcher.getClass.getClassLoader）这块特殊的机器码叫做启动类加载器。

  • 启动类加载器并不是java类，其他的加载器都是java类。

  • 启动类加载器是特定于平台的机器指令，它负责开启整个加载过程。

  • 启动类加载器会加载JRE正常运行所需要的基本组件，这包括java.util 和java.lang等
